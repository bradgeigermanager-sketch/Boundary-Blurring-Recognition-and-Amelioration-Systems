# Systems Dynamics – Drift Prevention Loops

Boundary drift emerges when complexity, ad‑hoc integrations, and unclear provenance erode domain separation. This document models the reinforcing and balancing loops that govern drift.

---

## Key Stocks

- **Boundary Clarity**  
  Strength of domain separation and ownership definition.

- **Signal Provenance Quality**  
  Accuracy and completeness of origin metadata.

- **Feedback Loop Alignment**  
  Degree to which loops remain confined to their intended domain.

- **Drift Pressure**  
  Accumulated forces pushing toward boundary blurring.

---

## Reinforcing Loop R1 – Provenance Reinforcement
Boundary Clarity ↑
→ Provenance Standards Adoption ↑
→ Signal Provenance Quality ↑
→ Correct Routing ↑
→ Feedback Loop Alignment ↑
→ Boundary Clarity ↑

This loop strengthens boundaries through improved provenance and routing.

---

## Balancing Loop B1 – Drift Detection & Correction
Drift Pressure ↑
→ Boundary Incidents ↑
→ Drift Detection ↑
→ Correction Actions ↑
→ Boundary Clarity ↑
→ Drift Pressure ↓

Incidents trigger detection and correction, restoring clarity.

---

## Reinforcing Loop R2 – Complexity‑Driven Drift
System Complexity ↑
→ Ad‑hoc Integrations ↑
→ Unmapped Cross‑Domain Loops ↑
→ Boundary Blurring ↑
→ Diagnostic Confusion ↑
→ More Ad‑hoc Fixes ↑

This loop drives drift unless constrained by architecture and standards.

---

## Goal

Use R1 and B1 to counteract R2, ensuring long‑term boundary stability.
